## Problem Statement

Multiple Telegram handlers use `context.Background()` instead of propagating the request context. This prevents proper cancellation when clients disconnect and can cause resource leaks.

**Locations**:
1. `internal/adapter/telegram/handler_cta.go:581`
2. `internal/adapter/telegram/handler_quant.go:448`
3. `internal/adapter/telegram/handler_quant.go:484`
4. `internal/adapter/telegram/handler_vp.go:422`

**Current Code Pattern**:
```go
ctx := context.Background()  // ← PROBLEM: no cancellation, no timeout control
// or
cmdCtx, cancel := context.WithTimeout(context.Background(), 60*time.Second)
```

---

## Impact

- **Severity**: Resource leaks, unbounded operations
- **Risk**: Operations continue after client disconnect; wasted compute
- **User Impact**: Slower response times under load; resource waste

## Implementation Notes

**Pattern to Use**:
```go
// Instead of:
ctx := context.Background()

// Use:
ctx := r.ctx  // request context from handler
// or
cmdCtx, cancel := context.WithTimeout(r.ctx, 60*time.Second)
defer cancel()
```

**Check Handler Context Availability**:
- Check if `handler` struct has access to request context
- May need to update handler method signatures to accept context
